# 第 3 章 系统总体设计

## 3.1 系统总体架构

整个系统可以分成四层来看：采集层、通信层、平台层和展示层。

- 采集层：ESP32-WROOM-32 + DHT11，负责采集温湿度。
- 通信层：BLE 负责首次配网，WiFi 负责后续业务通信。
- 平台层：Flask 接口、业务服务和 MySQL 数据库。
- 展示层：浏览器页面，负责设备管理、数据查看、告警和报告展示。

这样拆分的好处是职责比较清楚。设备只管采集和上传，后端只管存储和处理，页面只管展示和交互。联调时如果出问题，也比较容易判断是哪一层出了错。

我在实际开发里最明显的感受就是，物联网项目最怕“每层都沾一点，但谁都不清楚自己负责什么”。一旦接口字段没对齐，或者设备和页面都各写一套逻辑，后面排查会很痛苦。

## 3.2 功能模块设计

### 3.2.1 用户模块

用户模块负责注册、登录、退出和资料维护。这里虽然不是系统亮点，但它是其他模块的入口。

平台采用 Session 保存登录状态，用户登录后，设备管理、监测、控制和 AI 报告等页面都依赖这个会话信息。这样做比较轻，适合当前项目。

### 3.2.2 设备模块

设备模块的核心不是“显示一张设备表”，而是把用户和设备关系管起来。

平台会记录设备的 `device_uid`、`device_secret`、设备名称、固件版本、绑定用户和最近心跳时间。后面的数据、告警、命令和报告都围绕这张设备表展开。

在设计上，我还加了演示设备入口。原因很现实：有时真机不在身边，但页面功能和流程还是得继续调。

### 3.2.3 数据监测模块

数据监测模块分两块：最新值和历史数据。

最新值用来判断设备此刻状态，历史数据用来画曲线。页面上我没有做特别复杂的筛选，只保留了几个常用时间范围，因为这对答辩和日常查看都够用了。

相比整页都堆表格，曲线更容易看出温湿度有没有波动，设备是不是在持续上报。

### 3.2.4 控制模块

控制模块负责命令下发、设备拉取和回执处理。

我这里没有用实时推送，而是让设备轮询待执行命令。这个方案实时性一般，但胜在稳定、简单、好调。对于本科项目来说，把 `pending -> done / fail` 这条状态链做清楚，比一上来追求复杂通信方式更实际。

### 3.2.5 AI 分析模块

AI 模块负责把最近一段时间的数据摘要变成文字说明。它不是系统运行的前提，但能明显提升平台的完整度。

这一块我没有做自训练模型，而是调用第三方大模型接口，同时保留本地模板降级。这样做的重点不在“算法多先进”，而在于把现有业务数据和分析能力接上。

### 3.2.6 BLE 配网模块

BLE 配网模块是这次设计里比较关键的一块。它负责把设备从“还没联网”带到“已经在平台注册并开始上报”。

具体来说，浏览器扫描 ESP32 广播后，把 WiFi、`server_base` 和 `device_secret` 写进去，设备收到后保存参数，再去连网和注册。页面还能实时看到当前步骤。

这个模块之所以重要，是因为它直接影响系统第一次用起来是不是顺手。

## 3.3 后端分层设计

后端采用 `routes / service / dao / utils` 这种分层方式。

- `routes` 负责接收请求、检查参数、返回结果。
- `service` 负责组织业务流程。
- `dao` 负责和数据库交互。
- `utils` 放登录校验、统一响应和序列化之类的公共逻辑。

这套结构在开发前期看不出太大区别，但到后面加设备网关、AI 模块和告警逻辑时就很有用了。比如设备上报接口出了问题，我通常先看路由有没有接到数据，再看业务层有没有做阈值判断，最后看 DAO 是否写入成功。分层清楚以后，问题定位会快很多。

## 3.4 通信流程设计

### 3.4.1 BLE 配网流程

BLE 配网流程按实际执行顺序如下。

1. 用户在电脑浏览器打开配网页。
2. 浏览器扫描到 ESP32 的 BLE 广播设备。
3. 用户填写 WiFi、密码和服务器地址。
4. 浏览器把 JSON 配置写入 BLE 特征值。
5. ESP32 解析参数并保存到本地。
6. 设备连接 WiFi。
7. 设备调用平台注册接口。
8. 设备通过 BLE notify 把当前状态回传到页面。

这里最需要单独说明的一点，是浏览器打开页面的地址和设备真正访问的平台地址不一定相同。页面可以从 `127.0.0.1` 打开，因为这样能满足 Web Bluetooth 的安全上下文要求；但设备写入的 `server_base` 必须是局域网 IP 或公网地址。

### 3.4.2 数据上报流程

数据上报流程比较直接：

1. 设备采集温湿度。
2. 设备调用 `/api/device/data` 上传数据。
3. 平台校验设备身份。
4. 平台写入 `device_data` 表。
5. 平台刷新设备最近心跳时间。
6. 如果温度超阈值，再进入告警逻辑。

这条流程在实际联调里是最早打通的，也是后面所有功能的基础。只要这里不稳定，监测页、AI 报告和告警都没法往下做。

### 3.4.3 命令回执流程

命令闭环流程如下。

1. 用户在控制页面创建命令。
2. 平台把命令写入 `command_log`，状态为 `pending`。
3. 设备轮询 `/api/device/command`。
4. 设备执行命令后调用回执接口。
5. 平台更新命令状态和执行说明。

这条流程虽然不复杂，但它让平台从“只能看”变成了“还能管”。

## 3.5 数据库设计

系统数据库采用 MySQL。核心表主要有下面几张。

| 表名 | 主要作用 |
|------|----------|
| `user` | 存用户账号和联系信息 |
| `device` | 存设备身份、归属和在线状态 |
| `device_data` | 存设备上传的监测数据 |
| `command_log` | 存命令和回执结果 |
| `operation_log` | 预留操作日志 |
| `ai_report` | 存 AI 报告内容 |

### 3.5.1 用户表

用户表主要保存用户名、密码哈希、邮箱、手机号和创建时间等信息。这里邮箱字段比较重要，因为邮件告警要直接用到。

### 3.5.2 设备表

设备表是整个系统的中间枢纽。设备有没有绑定用户、最近有没有心跳、现在显示在线还是离线，都是从这张表里看出来的。

设备的唯一标识采用 `device_uid`，认证依赖 `device_secret`。实际联调时，只要这两个字段和固件里不一致，接口就会直接返回 403。

### 3.5.3 数据表

`device_data` 用来保存温度、湿度、光照和记录时间等信息。当前主要用到了温湿度，光照字段属于预留扩展。

考虑到监测页会频繁按时间范围查数据，所以在 `(device_id, recorded_at)` 上建立了索引。

### 3.5.4 命令与报告表

`command_log` 用来记录命令内容、执行状态和结果说明，`ai_report` 用来保存报告标题、摘要、正文和生成时间。

这两张表虽然不是最基础的业务表，但它们能体现系统比“普通采集程序”多做了什么。

## 3.6 关键接口设计

平台接口统一返回 JSON，核心字段就是 `code`、`msg` 和 `data`。这样前端和设备端处理起来都比较统一。

这里我没有在论文里保留太多模板化接口格式说明，重点只放在几类真正关键的接口上：

- 用户登录和资料接口。
- 设备绑定和列表接口。
- 设备注册、心跳、数据上报接口。
- 命令拉取和回执接口。
- AI 报告生成和查询接口。

实际联调时，一个小问题也暴露了接口统一的重要性。比如固件最初对返回内容的判断写得太死，只兼容一种 `code=0` 的格式，后来又补了对 `"code":0` 和 `"code": 0` 两种情况的兼容。这种问题单看很小，但会直接影响设备是否误判注册结果。

## 3.7 关键计算口径

本系统没有复杂算法，主要就是一些基础统计和判断。

- 统计最近一段时间的平均温度、最高温度、最低温度。
- 统计平均湿度。
- 判断温度是否超过告警阈值。
- 判断冷却时间内是否需要重复发邮件。

这些逻辑本身不复杂，但在系统里都是实打实要用到的。尤其是告警和 AI 报告，都是先基于这些统计结果，再往后走。

## 3.8 本章小结

本章主要说明了系统各模块怎么分、数据怎么流、数据库怎么建，以及后端为什么这样组织。把这些关系先讲清楚，后面的实现过程就不会显得很散。
