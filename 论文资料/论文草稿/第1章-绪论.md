# 第 1 章 绪论

## 1.1 研究背景

这次毕业设计做的是一个基于 Web 的物联网设备管理平台。课题看起来不算特别大，但真正做起来，里面把前端、后端、数据库、嵌入式、网络通信这些东西都串到了一起。

一开始我接触这个题目时，最直接的感受是：很多环境监测小系统都能采数据，但不好用。设备接入麻烦，换个 WiFi 还得改固件；页面能看到曲线，但是设备状态、告警、控制这些功能是断开的；真到了演示或者部署阶段，经常又会卡在地址配置、浏览器限制或者软硬件联调上。

所以我最后没有把重点放在某个单独算法上，而是把目标定成一条完整链路：ESP32 采数据，Flask 接口收数据，MySQL 存数据，网页看数据，还能做 BLE 配网、超温告警和 AI 分析。对本科毕设来说，把这些环节真正打通，比单独堆几个功能更有说服力。

## 1.2 研究意义

### 1.2.1 理论意义

这个项目把几门课里的内容放到同一个系统里验证了。前端不是单独做页面，后端也不是只写几个接口，硬件也不是只让串口打印数据，而是要一起工作。

从学习角度看，这种综合题目更能说明自己是不是真的理解了物联网系统的结构。比如设备为什么要注册，为什么要心跳，为什么 Web 页面能看到的数据不一定代表设备真的在线，这些问题只有把整条链路都做出来才会体会得比较深。

### 1.2.2 实践意义

这个平台的实际价值主要体现在下面几个方面。

- 设备能统一接入，不用每次都靠手改固件参数。
- 数据能在网页上直接看，适合演示，也方便排查问题。
- 温度超阈值时能主动提醒，不再只是被动看曲线。
- AI 报告把一段时间的数据转成文字说明，读起来比纯数字更直观。
- 后面如果要继续加光照、空气质量之类的传感器，也有继续改的基础。

另外，这个题目还有一个比较实际的意义，就是能把“看起来能跑”和“实际能用”区分开。我在开发过程中发现，很多问题不是代码能不能编译，而是系统能不能稳定联调。比如 `server_base` 填成 `127.0.0.1` 后，浏览器页面能打开，但 ESP32 连网以后根本访问不到服务器；再比如 Web Bluetooth 在 `localhost` 下能扫到设备，换成局域网 HTTP 页面就不行。这些都属于做完之后才真正踩到的坑。

## 1.3 国内外研究现状

查阅资料后可以看出，环境监测类系统已经很多了。国外的研究更强调架构规范、多参数采集和平台扩展能力，国内的研究更贴近具体应用场景，比如智能家居、农业大棚、便携式监测和行业预警。

### 1.3.1 国外研究现状

国外关于环境监测平台的研究起步比较早，很多工作已经不只停留在“采到数据”这一步，而是把采集、传输、存储和可视化都做成较完整的平台。比如有研究基于 Flask 搭建环境监测 Web 应用，说明轻量级后端框架已经可以较好支撑这类平台开发[1]；也有研究结合 ESP32 和开放式云平台构建低成本监测系统，重点放在远程监控和平台开放性上[2]；还有研究把模块化设计、多参数采集和实时服务结合起来，形成更完整的室内环境监测平台[3]。这类研究对本课题最大的参考，不是它们用了多复杂的模型，而是它们普遍比较重视系统闭环和指标验证。

不过国外不少方案偏向空气质量监测、智慧建筑或者较完整的工业场景，模块更多，系统规模也更大。像基于 ESP32-WROOM 的多参数室内环境监测系统，已经会把温湿度、光照甚至更多指标一起纳入，并对传输性能做量化验证[4]；另有研究结合 ESP32 与 Blynk 平台完成环境监测，也体现出国外在低成本硬件和可视化接入方面已经比较成熟[15]。对于本科毕设来说，直接照搬并不现实，更合适的做法是借鉴它们“平台化”和“模块化”的思路，再结合自己的开发条件做轻量实现。

### 1.3.2 国内研究现状

国内相关研究更强调落地。常见做法是“传感器 + 无线通信 + 上位机或 Web 平台”这条路线，重点放在监测展示、简单告警或者某个具体场景的应用上。比如有研究围绕智能家居环境监测展开设计，把温湿度、空气质量和光照等指标纳入统一监测范围[6]；也有研究强调便携式部署和多场景适配[5]；还有研究把物联网平台和居家安全场景结合起来，开始关注监测、告警和用户响应之间的联动关系[7]。

这些工作对我帮助比较大的一点，是让我确认这个题目不需要追求特别复杂的架构，关键是把设备接入、数据展示和业务功能做好。另外，在通信和硬件选型上，国内也已经有不少围绕 ZigBee、LoRa、STM32 和 ESP32 的实现研究[8-14]。从这些文献也能看出来，很多系统在论文里写得很完整，但真正涉及配网、部署和联调细节时，往往一带而过，这恰好也是我后面想补上的部分。

### 1.3.3 现有研究不足

结合文献和我自己的开发过程，现有研究里比较常见的不足主要有以下几类。

第一，很多系统对首次接入讲得不够细。论文里常写“设备联网后上传数据”，但设备到底怎么配网、改网络后怎么处理、普通用户怎么操作，往往写得比较笼统。

第二，平台层面容易停在“采集 + 图表”这一步。图能画出来当然重要，但如果没有告警、分析、控制或者状态反馈，平台的完整度还是不够。

第三，不少系统缺少真实的联调痕迹。比如接口字段改过几次、串口日志里遇到过什么问题、地址配置为什么会错，这些才是系统能不能落地的关键。

第四，测试章节容易模板化。很多内容是统一格式的“测试目标、步骤、结果”，但没有把实际现象写出来，读起来就比较空。

## 1.4 本文研究内容

围绕上面这些问题，本文主要做了以下工作。

1. 设计并实现一个基于 B/S 架构的物联网设备管理平台。
2. 使用 Flask 搭建后端接口，完成用户、设备、数据、命令和 AI 报告等模块。
3. 使用 MySQL 完成设备、时序数据和报告信息的存储。
4. 使用原生 HTML、CSS、JavaScript 和 Chart.js 实现页面交互与监测曲线展示。
5. 使用 ESP32-WROOM-32 与 DHT11 组成终端采集节点。
6. 实现基于 Web Bluetooth 的 BLE 配网流程。
7. 打通设备注册、心跳、数据上报、命令拉取和回执闭环。
8. 实现超温告警、邮件通知和 AI 报告分析。

和单纯做采集演示相比，本文更关注“整个平台能不能顺着用下来”。也就是说，设备不是只要能上传一条数据就算完成，而是要能接入、能展示、能提醒、能控制、能说明问题。

## 1.5 论文结构安排

第 1 章介绍课题背景、研究意义、研究现状和本文内容。

第 2 章分析系统需求，并说明方案选择过程。

第 3 章说明系统总体设计，包括架构、数据库和关键流程。

第 4 章结合实际开发过程介绍系统实现，重点写后端、前端、硬件接入、BLE 配网、告警和 AI 模块。

第 5 章介绍测试与联调过程，重点展示系统闭环是否真正打通。

第 6 章对全文进行总结，并说明目前的不足和后续可以继续优化的方向。
