# 第 6 章 总结与展望

## 6.1 全文工作总结

这次毕业设计最终完成的是一个基于 Web 的物联网设备管理平台。整个系统以前后端一体的方式实现，终端使用 ESP32-WROOM-32 和 DHT11 采集温湿度，平台使用 Flask 和 MySQL 完成数据处理与存储，页面部分使用原生 Web 技术和 Chart.js 完成展示与交互。

从结果来看，这个项目已经不只是“设备上传一条数据”这么简单，而是基本形成了完整流程：设备可以通过 BLE 配网接入平台，平台可以接收注册、心跳和监测数据，用户可以在网页上看设备状态和历史曲线，温度异常时可以触发邮件提醒，AI 模块还能根据最近一段时间的数据生成分析报告，控制页也能完成简单命令的下发和回执。

对我自己来说，这次毕设最有价值的部分，不是某一个功能单独写出来了，而是我第一次比较完整地把硬件、接口、数据库、页面和部署问题放到同一个项目里处理。前面很多课设都是单点实现，这次做下来以后，我对“一个系统为什么能跑起来”这件事理解得更具体了。

## 6.2 完成的主要工作

本文主要完成了以下几项工作。

1. 设计了平台整体架构，并划分出用户、设备、数据、命令、AI 和告警等模块。
2. 实现了 Flask 后端接口和 MySQL 数据库存储结构。
3. 实现了设备注册、心跳、数据上报和命令回执闭环。
4. 实现了监测页面、设备页面、控制页面、AI 报告页面和 BLE 配网页面。
5. 实现了基于 Web Bluetooth 的 BLE 配网流程。
6. 实现了温度阈值告警、邮件通知和 AI 报告分析。
7. 完成了本地联调和云端部署验证，并处理了一批实际问题。

这些“实际问题”我觉得也值得算进成果里。比如 GPIO12 导致启动不稳、`server_base` 填错导致设备注册失败、`.env` 修改后旧值残留、AI 请求过慢导致 worker 超时，这些都不是设计图里能直接看出来的，但它们确实决定了系统最后能不能稳定运行。

## 6.3 系统不足

虽然系统已经完成了预期的主要功能，但还有一些不足比较明显。

第一，当前使用的 DHT11 只是入门级传感器，精度和稳定性都有限，更适合演示，不太适合更严肃的监测场景。

第二，BLE 配网虽然做出来了，但 Web Bluetooth 对浏览器环境要求比较高。`localhost` 下比较顺利，普通局域网 HTTP 页面就会受限制，这一点在后续使用时还不够友好。

第三，AI 分析依赖第三方接口，网络状态不好时响应会变慢，虽然已经做了本地降级，但在线分析体验还是会受影响。

第四，当前监测维度还比较少，主要围绕温湿度展开，数据类型不够丰富。

第五，远程控制采用轮询方式，优点是简单，但实时性一般，如果后续设备数量再增加，体验上会有局限。

## 6.4 后续展望

如果后面继续完善，我认为可以从下面几个方向继续做。

第一，补充更多传感器，比如光照、空气质量、烟雾或者人体检测模块，让数据维度更丰富。

第二，把平台和配网页统一放到 HTTPS 环境下，减少 Web Bluetooth 的环境限制。

第三，优化设备侧缓存和补传机制，进一步提高弱网环境下的稳定性。

第四，把命令通信从当前轮询方式升级到更实时的方案，比如 MQTT 或 WebSocket。

第五，继续细化 AI 分析逻辑，不只是做统计描述，还可以加入更细的异常趋势判断。

第六，补充多设备、多用户场景下的管理细节，让平台更接近真实应用。

## 6.5 本章小结

本章对全文工作做了总结，也把当前系统的短板和后续方向说清楚了。整体来看，这个项目已经完成了一套能实际演示、能真实联调、也有继续扩展空间的物联网平台原型。
