# 术语表与专业解释

## 1. 核心术语

### B/S 架构

Browser/Server 架构，即浏览器/服务器架构。用户通过浏览器访问系统，业务逻辑主要部署在服务器端。

### Flask

Python 语言中的轻量级 Web 框架，适合快速构建中小规模 Web 应用。

### RESTful 接口

一种面向资源的接口设计风格，通常基于 HTTP 方法实现资源的查询、创建、修改和删除。

### PyMySQL

Python 连接 MySQL 数据库的常用库，用于执行 SQL 和管理数据库访问。

### Chart.js

轻量级前端图表库，用于在网页中绘制折线图、柱状图等可视化图形。

### ESP32-WROOM-32

一款集成 WiFi 和蓝牙功能的微控制器模块，适用于物联网终端开发。

### DHT11

一种常见的温湿度传感器，可采集环境温度和湿度信息，适合教学和基础实验场景。

### BLE

Bluetooth Low Energy，低功耗蓝牙。常用于短距离低功耗通信，本项目中用于设备配网。

### Web Bluetooth

浏览器提供的蓝牙访问能力，允许网页与蓝牙设备进行交互，但受安全上下文限制。

### `server_base`

设备侧需要访问的平台基础地址，例如 `http://192.168.x.x:5000` 或公网地址。它必须是设备真实可访问的地址，不能简单写为 `127.0.0.1`。

### `device_uid`

设备唯一标识，用于区分不同终端。

### `device_secret`

设备身份校验密钥，用于设备访问平台接口时进行鉴权。

### 心跳机制

设备按固定时间周期向服务器发送在线状态请求，用于维持平台侧的设备在线信息。

### 阈值告警

当监测值超过设定阈值时触发的告警逻辑。本项目中主要指温度超阈值告警。

### 冷却时间

在同一设备连续触发告警时，为避免重复发送通知而设置的最小时间间隔。

### AI 场景化分析

根据设备统计数据和使用场景，由第三方大模型或本地模板生成的辅助说明性分析文本。

## 2. 建议统一表述

- 不写“Java Web”，统一写“Python + Flask Web 平台”
- 不写“算法模型训练”，统一写“第三方大模型辅助分析”
- 不写“智能预测系统”，除非后续确实补做预测算法
- 不写“高精度温度检测”，DHT11 更适合写“基础环境监测”

## 3. 答辩时可直接使用的解释口径

- 为什么选 Flask：轻量、灵活、适合本科毕设规模，能快速完成接口和业务实现。
- 为什么选 ESP32：单芯片集成 WiFi 和 BLE，既能联网又能配网，适合本项目。
- 为什么做 BLE 配网：减少手工改固件参数的麻烦，体现系统完整性。
- 为什么加入 AI：增强数据报告的可读性，但不把 AI 作为论文唯一核心创新点。
