# 第 5 章 系统测试与结果分析

## 5.1 测试目标

这一章的测试重点不是跑出多漂亮的性能指标，而是验证系统到底有没有真正形成闭环。换句话说，我最关心的不是某个接口单独返回了 200，而是设备接入、数据上报、页面展示、告警、AI 报告和命令回执这些环节能不能连起来工作。

因此，本章主要验证三件事：

1. 各模块单独使用时是否正常。
2. 关键链路串起来以后是否稳定。
3. 遇到常见异常时，系统能不能给出合理反馈。

## 5.2 测试环境

测试环境以我实际开发和联调使用的环境为主，重点不是把参数写得特别满，而是说明系统是在什么条件下跑通的。

| 项目 | 实际情况 |
|------|----------|
| 开发系统 | Windows 本机环境 |
| 后端框架 | Flask |
| 数据库 | MySQL 8.0 |
| 前端运行方式 | Chrome / Edge 浏览器访问本地页面 |
| 终端硬件 | ESP32-WROOM-32、DHT11 |
| 终端接线 | DHT11 数据脚接 GPIO27 |
| 配网方式 | Web Bluetooth + BLE |
| 部署验证 | 本地运行，后期补做过 Ubuntu + Nginx + Gunicorn 联调 |

这里有几个环境细节需要单独记一下。

第一，ESP32 板子用的是 CH340 串口方案，前期如果驱动没装好，Arduino IDE 根本识别不到串口。第二，Type-C 线也不是随便拿一根就行，我一开始就遇到过“能供电但不能传数据”的线。第三，BLE 配网页在 `localhost` 下能正常工作，换成普通局域网 HTTP 页面就会受浏览器策略限制。

## 5.3 测试方法

本系统测试没有完全照搬标准模板，而是按我实际开发时的排查顺序来做，主要分成三种方式。

第一种是页面和接口的基本功能测试，比如能不能登录、设备列表能不能显示、历史数据能不能查出来。

第二种是接口联调测试，重点看 ESP32 和后端的注册、心跳、上报、命令回执这些流程是否打通。

第三种是场景测试，也就是把配网、数据展示、告警、AI 报告和控制命令连续操作一遍，看看整条链路有没有断点。

这种测试方法更贴近实际，因为我开发时也不是按“模块一、模块二、模块三”孤立推进的，而是经常边写边联调，哪里断了就回头查哪里。

## 5.4 关键功能测试

### 5.4.1 用户功能测试

| 编号 | 我实际测试的内容 | 结果 |
|------|------------------|------|
| U-01 | 新用户注册后再登录 | 可以正常进入首页 |
| U-02 | 未登录直接访问设备相关接口 | 后端返回未登录提示 |
| U-03 | 修改个人资料并保存邮箱 | 页面能显示更新后的信息 |

这一部分虽然不算复杂，但它是后续所有业务的入口。如果登录态有问题，后面设备列表、AI 报告和控制页面都会跟着出问题。

### 5.4.2 设备接入与数据测试

| 编号 | 我实际测试的内容 | 结果 |
|------|------------------|------|
| D-01 | ESP32 调用注册接口 | 平台成功生成或识别设备记录 |
| D-02 | 设备周期发送心跳 | 设备在线状态能更新 |
| D-03 | 设备上传温湿度数据 | `device_data` 表写入成功，页面可见 |
| D-04 | 监测页按时间范围查历史 | 曲线能正常切换和刷新 |

这部分联调时有两个问题最典型。

一个是 GPIO12 启动干扰。刚开始传感器数据脚接在 GPIO12，上电和下载都不稳定，后来改到 GPIO27 才正常。

另一个是设备注册结果误判。固件最早对返回内容判断得太死，接口虽然成功了，设备却没识别出来，后面兼容 `"code":0` 和 `"code": 0` 两种格式后，这个问题才消失。

### 5.4.3 BLE 配网测试

| 编号 | 我实际测试的内容 | 结果 |
|------|------------------|------|
| B-01 | 浏览器扫描 BLE 设备 | 能扫到 ESP32 配网广播 |
| B-02 | 下发 WiFi 和 `server_base` | 设备能收到并写入配置 |
| B-03 | 观察配网状态回传 | 页面能看到连接 WiFi 和注册进度 |

BLE 配网测试是我整套系统里最想保留下来的部分，因为这里最容易体现“不是照着模板拼出来的”。

实际联调时，我遇到过下面几个很具体的问题。

第一，页面从 `127.0.0.1` 打开时蓝牙能扫到，但如果把 `127.0.0.1` 也写进设备的 `server_base`，设备注册一定失败，因为设备访问到的是它自己。

第二，页面从局域网地址打开时，Web Bluetooth 可能直接不可用。这不是 Flask 有问题，而是浏览器把普通局域网 HTTP 判成了非安全上下文。

第三，如果数据库里已经有同一个 `device_uid` 的旧记录，而 `device_secret` 又不一致，设备会直接收到 403。这个问题第一次碰到时，我还以为是接口没开，后来才发现是旧数据没清干净。

### 5.4.4 告警与 AI 测试

| 编号 | 我实际测试的内容 | 结果 |
|------|------------------|------|
| A-01 | 上传高于阈值的温度 | 页面出现告警状态，邮件能发送 |
| A-02 | 连续触发高温 | 冷却时间内不会一直重复发 |
| A-03 | 手动发起 AI 分析 | 报告可以生成并入库 |
| A-04 | 模拟 AI 接口异常 | 系统能回退到本地模板 |

这一部分测试最能说明平台是不是“只会显示数据”。

邮件告警这边，我前面调了好几次才稳定。一个坑是 `.env` 改完以后进程没真正读到新值，导致发件人地址一直错；另一个坑是如果不加冷却时间，只要设备持续高温，邮箱会一直收到重复邮件。

AI 模块这边，云上部署后还遇到过请求慢导致 gunicorn worker timeout 的情况。后来把超时参数调大，并保留本地降级逻辑，体验才正常。

### 5.4.5 命令闭环测试

| 编号 | 我实际测试的内容 | 结果 |
|------|------------------|------|
| C-01 | 在页面创建命令 | 命令成功写入待执行队列 |
| C-02 | 设备轮询拉取命令 | 能拿到待执行命令 |
| C-03 | 设备执行后提交回执 | 页面能看到状态变化 |

命令闭环测试的意义在于，平台不再只是“看设备”，而是开始“管设备”。虽然当前命令比较简单，但至少已经形成了下发、执行、回执这条链。

## 5.5 关键联调过程

如果只看上面的单项测试，系统已经能说明大部分功能。但真正能体现完成度的，还是整条链路连续跑通。

我实际联调时，基本按下面这个顺序操作：

1. ESP32 上电，进入 BLE 广播。
2. 浏览器打开配网页并扫描到设备。
3. 输入 WiFi、密码和服务器地址后发送配置。
4. 设备连接 WiFi，并向平台注册。
5. 平台设备列表出现该设备，状态逐渐变成在线。
6. 设备开始周期上传温湿度数据。
7. 监测页显示最新值和历史曲线。
8. 把阈值调低或模拟高温数据，触发邮件告警。
9. 在 AI 页面发起分析请求，生成报告。
10. 在控制页下发命令，设备执行后回执结果。

这条流程里，最能证明系统成型的不是某一张图，而是前后能对上。

比如配网页显示“已连接 WiFi”时，串口里能看到对应日志；设备列表显示在线时，数据库里也能查到新心跳；监测页画出曲线时，`device_data` 表里能看到相同时间段的数据；控制页显示命令完成时，串口里也能看到执行记录。

## 5.6 测试结果分析

### 5.6.1 功能完整性分析

从功能完整性看，系统已经完成了用户登录、设备接入、数据上报、曲线展示、告警提醒、AI 报告和命令闭环这些主要环节。对本科毕设来说，这样的功能覆盖已经比较完整，不再只是一个单独的采集程序或者展示页面。

### 5.6.2 稳定性分析

从稳定性看，本地联调阶段设备可以持续上报，监测页也能稳定刷新。数据库中能够连续看到真实记录，说明采集、传输和入库这条链路是稳定的。

当然，这种稳定不是“完全没问题”，而是“问题能被定位并解决”。像 GPIO12 干扰、`server_base` 填错、`.env` 残留、AI 请求超时这些情况，都是在联调过程中真实出现过的。

### 5.6.3 工程实用性分析

从工程实用性看，BLE 配网和邮件告警是两个比较明显的加分项。

BLE 配网把首次接入从“改代码、重烧录”变成了“网页配置”，对演示和后续使用都方便很多。邮件告警则让平台从被动看曲线，变成了遇到异常能主动提醒。AI 报告虽然不是硬需求，但也让监测结果更容易解释。

### 5.6.4 现有不足分析

不足也比较明确。

- DHT11 精度一般，数据维度偏少。
- Web Bluetooth 受浏览器环境限制比较明显。
- AI 报告仍依赖外部服务。
- 控制命令采用轮询，实时性一般。

这些不足不会否定当前系统已经完成的内容，但后面如果继续深入，就得围绕这些地方继续补。

## 5.7 可继续补充的测试数据

如果后面还要继续压低 AI 痕迹、增强真实性，我建议在这一章再补几类你最容易拿到的真实记录。

1. 连续运行 30 分钟或 1 小时的上报现象。
2. 3 到 5 次 AI 报告生成耗时。
3. 邮件告警从触发到收到的大概时间。
4. 一次命令下发到回执完成的时间范围。
5. 断网后恢复补传的记录。

这些数据不用特别复杂，只要真实、能截图、能和正文对上，就比大段模板分析更有用。

## 5.8 本章小结

本章重点不是证明“所有测试都通过”，而是说明系统从设备接入到平台管理这条链路确实跑通了。相比统一模板式测试描述，这种带着真实问题和排查过程的写法，更能反映系统的实际完成度。
