# 第 4 章 系统实现

## 4.1 后端模块实现

### 4.1.1 Flask 应用初始化

后端采用 Flask 应用工厂方式组织，入口文件是 `run.py`。应用启动时会加载环境变量、注册蓝图、初始化跨域配置，然后把各类接口挂到统一的路由前缀下。

这么做的好处是后面加模块时不容易乱。实际开发到中后期时，我陆续补了 AI、设备网关、蓝牙配网辅助和告警逻辑，如果一开始就把代码全堆在一个文件里，后面基本没法维护。

另外，这里还踩过一个环境变量的坑。项目最开始改过 `.env` 之后，程序有时还是读旧值，后来排查发现是 `python-dotenv` 默认不会覆盖同名进程变量，最终通过 `load_dotenv(override=True)` 才把这个问题解决掉。SMTP 发件人地址残留就是在这里暴露出来的。

### 4.1.2 用户认证模块实现

用户认证模块负责注册、登录、退出和资料修改。平台使用 Session 保存登录状态，前端请求统一携带 Cookie。

我没有把这部分做成前后端完全分离的 Token 方案，原因主要是当前项目规模不大，而且页面本身就是浏览器访问为主，Session 更省事。对毕设来说，这种实现足够清楚，也更容易把登录校验逻辑讲明白。

### 4.1.3 设备管理模块实现

设备管理模块负责设备列表展示、绑定、改名、删除和阈值设置。每台设备会保存唯一标识、密钥、当前归属用户、最近心跳时间和基础配置信息。

这块实现里有一个比较实用的设计，就是演示设备。因为我前期很多页面不是一直拿真机调的，演示设备让我能先把页面、数据库和报告流程走通，再回过头接真实硬件。

### 4.1.4 设备网关模块实现

设备网关模块是后端和 ESP32 对接的核心。当前已经实现的接口包括：

- 设备注册接口
- 心跳接口
- 数据上报接口
- 命令拉取接口
- 命令回执接口

设备首次接入时会调用注册接口，把 `device_uid`、`device_secret`、固件版本和名称传上来。后面正常运行阶段则周期性发心跳和数据。

这一块联调时最明显的经验是，设备侧接口一定要先做身份校验，再做业务处理。否则设备密钥错了、字段错了，数据库里还照样写进脏数据，后面更难查。

## 4.2 数据处理与存储实现

### 4.2.1 数据上报处理

ESP32 连网后会周期性读取 DHT11 数据，并通过 HTTP 向平台提交。后端收到后先校验设备身份，再写入 `device_data`，最后同步刷新设备状态。

我在实现时还保留了批量上传能力，是为了给离线缓存补发做准备。因为真实网络不可能一直稳定，如果设备断网几分钟，恢复后能把缓存数据补回来，平台上的曲线会更完整。

### 4.2.2 历史数据查询

监测页支持按时间范围查看历史数据。后端根据设备编号和起止时间查数据，再按前端需要的格式返回。

这里看起来像个普通查询，其实对体验影响挺大。历史查询一慢，监测页就会卡；数据格式一不统一，前端图表就容易画错。所以我后面把常用时间范围固定下来，也是为了减少无意义的复杂输入。

### 4.2.3 统计结果生成

AI 模块不是把所有原始数据都直接塞给模型，而是先统计最近一段时间的平均值、最大值、最小值和样本数量，再把这些结果拿去生成报告。

这样做有两个好处。一个是减轻请求体长度，另一个是让报告内容更聚焦。否则原始数据太散，模型生成的文字也容易飘。

## 4.3 前端页面实现

### 4.3.1 页面结构实现

前端采用原生 HTML、CSS 和 JavaScript 实现，主要页面包括登录、注册、首页概览、设备管理、实时监测、远程控制、AI 报告、个人中心和 BLE 配网页面。

我没有上前端框架，主要是因为当前页面数量虽然不少，但每页逻辑都比较清楚，用原生方式更轻，也更容易和后端接口一一对应。

### 4.3.2 前后端交互实现

前端通过统一封装的 `fetch` 方法访问接口，请求里带会话信息，返回结果统一按 `code / msg / data` 处理。

这么做的最大好处是少写重复逻辑。尤其在未登录跳转、错误提示和接口异常提示这几块，如果每个页面各写一套，后面改起来很麻烦。

### 4.3.3 图表展示实现

监测页面使用 Chart.js 绘制温湿度曲线。相比纯表格，曲线更适合看数据有没有连续变化，也更容易发现异常波动。

在调这部分时，我主要关注两件事：一是图能不能跟着时间范围正常切换，二是页面看到的曲线是不是和数据库里的真实记录对得上。因为如果前端格式转换有问题，页面上画出来的图看着正常，实际上可能已经错位了。

### 4.3.4 AI 报告页面实现

AI 报告页面负责展示报告列表、详情和删除操作。用户可以选择场景发起分析，生成后的报告保存到数据库，后面还能继续查看。

这一块在页面上不难，真正麻烦的是后端耗时不稳定。云上联调时出现过 AI 分析超时，前端拿到的不是 JSON，而是超时页面，最后才顺着日志定位到 gunicorn worker timeout。

## 4.4 ESP32 终端实现

### 4.4.1 硬件组成与接线

终端硬件使用 ESP32-WROOM-32 和 DHT11。DHT11 的数据脚最终接到了 GPIO27。

这个接线不是随便选的。前面我试过 GPIO12，结果下载和启动都不稳定，后来查资料并结合实测，确认它属于容易引发启动干扰的引脚，改到 GPIO27 之后问题才消失。这也是我在论文里想保留下来的一个真实联调细节。

### 4.4.2 数据采集实现

设备定时读取 DHT11 的温湿度值，然后组织成 JSON 上传到平台。同时，设备还会定时发送心跳，告诉平台自己还在线。

真机联调时，串口已经可以持续输出温湿度读数，后端也能正常返回 `code=0`。数据库和监测页里都能看到真实设备记录，这说明采集到展示这一段已经打通。

### 4.4.3 设备控制实现

当前设备支持的控制命令包括重启、修改上传周期、打开传感器和关闭传感器。设备会轮询平台拉取命令，执行后再提交回执。

这块虽然简单，但很适合说明平台已经不是“单向上传数据”的结构了。

## 4.5 BLE 配网实现

### 4.5.1 设计思路

BLE 配网的目标，就是把“设备第一次联网”这件事从改代码、重新烧录，变成网页上几步就能完成的操作。

页面负责扫描蓝牙设备、填写参数、发送配置，ESP32 负责接收参数、写入本地存储、连接 WiFi 并完成平台注册。

### 4.5.2 页面与固件协同

浏览器页面和固件之间约定了统一的 Service UUID 和 Characteristic UUID。页面把 WiFi 信息和服务器地址按 JSON 写入 BLE 特征值，设备则通过 notify 把当前阶段状态回传回来。

这一块联调时比较顺利，但也出现过两个比较典型的问题。

第一个问题是浏览器环境限制。`localhost` 下能用的蓝牙页面，换成 `http://192.168.x.x` 就可能直接失效，不是后端问题，而是 Web Bluetooth 的安全上下文限制。

第二个问题是服务器地址填写。前端页面在本机打开时很容易让人顺手填 `127.0.0.1`，但设备连上 WiFi 后访问不到这个地址，所以后面我又加了 `lan-hint` 接口和前端拦截逻辑，尽量避免再填错。

### 4.5.3 关键约束说明

BLE 配网最终能用，靠的不只是 UUID 对上，还包括几个约束条件一起满足。

- 浏览器要支持 Web Bluetooth。
- 页面最好在 `localhost` 或 HTTPS 环境下打开。
- 下发给设备的 `server_base` 不能是回环地址。
- 数据库里已有设备时，`device_secret` 必须和固件一致。

这些约束如果不提前说明，使用的人很容易觉得是“蓝牙坏了”或者“后端没启动”，其实问题常常出在地址和环境上。

## 4.6 AI 分析模块实现

### 4.6.1 分析流程实现

AI 分析流程分三步。

1. 先统计最近一段时间的数据摘要。
2. 再根据场景拼出提示词，请求第三方 AI 接口。
3. 最后把结果保存到数据库，供页面查看。

这个模块我最开始想得很简单，后来真正部署到云服务器后才发现，外部接口慢一点，整条链路就会受影响。也正因为这样，本地模板降级才不是“可有可无”的备选，而是必须要有的兜底。

### 4.6.2 场景化提示实现

系统目前支持机房、家庭、工厂和自定义场景。不同场景下，提示词会强调不同重点，比如机房更看重温度稳定性，家庭更偏向舒适度，工厂更关注风险和连续运行。

这样做的目的是让报告内容别太空。即使底层还是同一批统计数据，场景不同，说明方式也应该不一样。

### 4.6.3 本地降级实现

当在线 AI 服务不可用时，平台会自动切到本地模板，基于已有统计结果生成一份基础报告。

这部分在云上联调时确实派上过用场。因为实际出现过 AI 接口响应慢、worker 超时的问题，如果没有降级，用户点击分析后只能看到报错，体验会很差。

## 4.7 告警通知实现

### 4.7.1 阈值告警实现

每台设备都可以单独配置温度阈值。设备上传数据后，后端读取该设备阈值并做比较，超出后进入告警流程。

我把阈值设计成设备级，而不是用户全局级，主要是因为不同设备可能放在不同环境里，用同一个温度线不合理。

### 4.7.2 邮件通知实现

邮件通知通过 SMTP 发送，邮件内容包含设备名称、当前温度、阈值和简单处理建议。

这部分调试时也踩过坑。前面 SMTP 配置改了 `.env` 但程序没重新读进去，导致一直用旧发件人地址；后面解决完环境变量覆盖问题，再加上测试邮件接口，才算真正稳定下来。

### 4.7.3 告警状态展示实现

除了发邮件，平台还会记录最近一次告警状态，比如发送成功、发送失败、未配置邮箱或者还在冷却时间内。

这个状态展示看起来是小功能，但对调试非常有用。否则页面上只知道“没收到邮件”，却不知道是阈值没触发、邮箱没配好，还是被冷却机制拦住了。

## 4.8 本章小结

本章从后端、前端、终端、BLE 配网、AI 和告警几个方面说明了系统是怎么一步步做出来的。相比只写“实现了某某功能”，我更想保留这些联调细节，因为它们更能体现系统到底是怎么落地的。
