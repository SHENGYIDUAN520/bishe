## 一、仓库目录结构（必须遵守）

```
bishe/
├── 开发需求.md              # 业务需求与课题对应（范围级）
├── 开发文档.md              # 本文件：工程规范 + API + 库表 + 目录约定
├── README.md                # 安装、启动、环境变量（操作级）
├── requirements.txt         # Python 依赖及固定版本号
├── .env.example             # 环境变量模板（勿提交真实密钥）
├── run.py                   # 开发环境入口：创建 app 并启动
├── app/
│   ├── __init__.py          # 应用工厂 create_app、注册蓝图、CORS
│   ├── config.py            # 配置类（从环境变量读取）
│   ├── db.py                # PyMySQL 连接获取 / 上下文管理（仅 DAO 层使用）
│   ├── utils/
│   │   ├── response.py      # 统一 JSON 响应封装
│   │   └── auth_helper.py   # 登录会话、设备签名校验等（随开发补充）
│   ├── routes/              # 路由层：参数校验、调用 service、返回 JSON
│   ├── service/             # 业务逻辑层：事务边界、调用多个 DAO
│   └── dao/                 # 数据访问层：仅含 SQL + PyMySQL，禁止写业务判断
├── static/                  # 前端静态资源（CSS/JS/图片）
│   ├── css/
│   ├── js/
│   │   └── api.js           # 统一 fetch 封装（credentials、错误提示）
│   └── pages/               # 各功能页面 HTML（中文界面）
├── sql/
│   └── init.sql             # 建库建表、索引、可选初始数据
└── firmware/                # ESP32 Arduino 工程（后续补充，与硬件接口章节一致）
```

**分层职责**

- **routes**：解析 `request`，必填校验，调用 `service`，**不写 SQL**。
- **service**：组合业务规则、调用 `dao`，**不直接操作 `request` 对象**（可传参）。
- **dao**：`execute` / `fetch` 等，**必须使用参数化占位符**（`%s`），禁止字符串拼接 SQL。

## 四、统一接口规范

### 4.1 HTTP 与路径前缀

- 浏览器调用的业务接口：**`/api`** 前缀（REST 风格）。
- 设备端（ESP32）上报与拉取指令：**`/api/device`** 下独立子路径（见 4.4），与 Web 会话分离，使用**设备身份校验**。

### 4.2 统一响应体

所有 JSON 接口返回：

```json
{
  "code": 0,
  "msg": "操作成功",
  "data": null
}
```

| code | 含义 |
|------|------|
| 0 | 成功 |
| 400 | 参数错误 / 业务校验失败 |
| 401 | 未登录或登录失效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

- `data` 成功时为对象或数组；失败时可为 `null` 或附带简要字段。
- **异常必须捕获**，向客户端返回友好中文 `msg`，**禁止**堆栈信息直接暴露给前端。

### 4.3 用户会话（Web）

- 登录成功后使用 **Flask session**（cookie）维持登录态。
- **设备管理、数据、控制、AI 报告等接口**：除登录/注册及文档明确标注的公开接口外，**必须先校验已登录**，否则 `code=401`。

### 4.4 设备身份校验（硬件）

- 每台设备有唯一 **`device_uid`**（如芯片 MAC 大写无冒号）与 **`device_secret`**（平台生成或注册时返回）。
- 设备调用接口时：请求体或查询参数携带 `device_uid` + `device_secret`（或后续升级为 HMAC，本阶段以明文 secret 传输须在 HTTPS 或内网演示环境说明）。
- 校验失败：`code=403`，`msg` 说明非法设备。

## 五、数据库设计（与 sql/init.sql 一致）

**字符集**：utf8mb4；**引擎**：InnoDB。

| 表名 | 用途 |
|------|------|
| user | 用户 |
| device | 设备主数据与归属用户 |
| device_data | 时序传感数据 |
| command_log | 指令下发与执行结果 |
| operation_log | 用户关键操作审计 |
| ai_report | AI 分析报告 |

**关键字段约定**

- 用户表物理名为 **`user`**，SQL 中一律写作 `` `user` ``（保留字）；`password_hash` 长度 **VARCHAR(512)**，兼容 Werkzeug 多种算法。
- 各表含 `created_at`、`updated_at`（datetime，应用层或 DB 默认值维护）。
- `device.device_uid` 唯一；`device.device_secret` 存哈希或密文（演示阶段可与文档同步说明）。
- `device_data` 对 `(device_id, recorded_at)` 建索引以支持时间范围查询。

详细字段见 `sql/init.sql`；**改表必须同时改 init.sql 与本节文字**。

## 七、硬件与后端契约（摘要）

- 配网流程：BLE 收 SSID/密码 → 断 BLE → WiFi 连接 → 调用 **`POST /api/device/register`** → 正常运行。
- 周期性：**heartbeat**、**data** 上传；轮询或使用长轮询 **`GET /api/device/command`**（后续可演进 WebSocket，毕设阶段以简单方式为主）。
- 离线缓存 100 条补发：设备端固件实现；服务端 **`POST /api/device/data`** 支持批量数组。

### 7.1 当前联调固件（已提供）

- 路径：`firmware/esp32_dht11_gateway/esp32_dht11_gateway.ino`
- 传感器：DHT11，数据引脚 `GPIO27`
- 上报周期：数据约 10 秒一次，心跳约 30 秒一次
- 注意：`SERVER_BASE` 必须填写 Flask 电脑的局域网 IP，不能用 `127.0.0.1`
- BLE 配网固件（真机）：`firmware/esp32_ble_provisioning_gateway/esp32_ble_provisioning_gateway.ino`（配合 `static/pages/ble_setup.html`）

### 7.2 当前联调状态（2026-04-06）

- 已完成真机验证：ESP32 + DHT11(GPIO27) 能持续调用 `register/data/heartbeat` 且返回 `code=0`。
- 数据库已记录真实设备与实时数据，`monitor` 页面可展示曲线与最新值。
- 已知经验：GPIO12 属于启动相关引脚，传感器接入会影响稳定下载，建议统一使用 GPIO27。
- 设备绑定说明：绑定使用 `device_uid + device_secret`；固件密钥变更后需与数据库保持一致。

### 7.3 BLE 配网（Web Bluetooth）协议约定（2026-04-07）

**目标**：浏览器通过 Web Bluetooth 下发 WiFi 配置，ESP32 写入 NVS 后连网并调用 `POST /api/device/register`，同时把进度通过 notify 回传，便于答辩演示「配网->注册->平台展示」闭环。

**对应文件**

- 网页：`static/pages/ble_setup.html`
- 固件：`firmware/esp32_ble_provisioning_gateway/esp32_ble_provisioning_gateway.ino`

**BLE UUID（必须一致）**

- Service UUID：`9b5c0001-3f38-4f4a-9c2f-2b6c2e4b5c01`
- Config（WRITE）Characteristic UUID：`9b5c0002-3f38-4f4a-9c2f-2b6c2e4b5c01`
- Status（READ/NOTIFY）Characteristic UUID：`9b5c0003-3f38-4f4a-9c2f-2b6c2e4b5c01`

**下发 JSON（Config 特征值，UTF-8）**

```json
{
  "ssid": "Lab_IoT",
  "password": "12345678",
  "server_base": "http://192.168.188.19:5000",
  "device_secret": "esp32-ble-demo-secret"
}
```

- `ssid`：必填
- `password`：可选（开放网络可空）
- `server_base`：可选，不填则固件使用默认值
- `device_secret`：可选，不填则固件使用默认值

**状态回传（Status 特征值 notify/read，UTF-8 JSON 文本）**

- 示例：`{"stage":"wifi","ok":true,"msg":"WiFi 已连接","ip":"192.168.188.50","rssi":-48,"device_uid":"ESP32-...."}`
- `stage` 取值示例：`boot` / `ble` / `prov` / `wifi` / `register`

**安全与限制说明（毕设答辩口径）**

- Web Bluetooth 需要 **Windows 桌面 Chrome/Edge** 且页面在 **HTTPS 或 localhost**（安全上下文）。
- 设备端使用 `device_uid + device_secret` 明文演示：仅适用于内网/实验环境；可在论文中说明后续可升级为 HTTPS/HMAC。

### 7.4 超温告警链路（2026-04-10）

- 告警配置来源：用户联系信息来自 ` /api/user/profile `（邮箱、手机号）；阈值来自设备级设置 `PUT /api/devices/<id>/alert-threshold`。
- 触发时机：设备上报 `POST /api/device/data` 时，若温度超过阈值，后端尝试发邮件（含本地 AI 建议文本）。
- 限流策略：同一用户同一设备默认 `TEMP_ALERT_COOLDOWN_MINUTES=10` 分钟内不重复发。
- 前端可视化：`monitor` 页面展示最近一次邮件告警状态；并在新数据超阈值时弹窗展示 AI 建议。
- 运维建议：SMTP 相关环境变量必须正确配置；修改 `.env` 后重启进程。
