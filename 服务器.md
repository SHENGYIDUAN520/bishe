# 服务器运维记录（Ubuntu + systemd + iot-flask）

> 最近更新：2026-04-13  
> 项目目录：`/root/bishe`  
> 服务名：`iot-flask`

---

## 1) 日常更新代码后重启

```bash
cd ~/bishe
git pull
source .venv/bin/activate
pip install -r requirements.txt
sudo systemctl restart iot-flask
```

## 2) 查看状态与日志

```bash
sudo systemctl status iot-flask
journalctl -u iot-flask -f
```

常用命令：

```bash
sudo systemctl stop iot-flask
sudo systemctl start iot-flask
sudo systemctl restart iot-flask
sudo systemctl is-active iot-flask
sudo systemctl enable iot-flask
```

如果改过 `/etc/systemd/system/iot-flask.service`：

```bash
sudo systemctl daemon-reload
sudo systemctl restart iot-flask
```

## 3) 当前生效服务配置（关键）

```ini
[Unit]
Description=IoT Flask (gunicorn)
After=network.target mysql.service

[Service]
User=root
Group=root
WorkingDirectory=/root/bishe
Environment="PATH=/root/bishe/.venv/bin"
ExecStart=/root/bishe/.venv/bin/gunicorn -w 2 -t 180 -b 127.0.0.1:8000 run:app
Restart=always

[Install]
WantedBy=multi-user.target
```

## 4) AI 分析超时排障结论（2026-04-13）

- 现象：`/api/ai/analyze` 在云上偶发慢响应，出现 fallback 或 worker timeout。
- 结论：方舟接口可连通（`curl` 可达，带鉴权请求可返回 200），但完整分析请求在云上存在慢响应窗口。
- 当前策略：
  - `.env` 支持 `AI_HTTP_TIMEOUT` 与 `AI_MAX_RETRIES`
  - gunicorn 增加 `-t 180`，避免被默认 30 秒提前杀 worker
- 推荐参数（云上）：
  - `AI_HTTP_TIMEOUT=30~90`
  - `AI_MAX_RETRIES=1`
